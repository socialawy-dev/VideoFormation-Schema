## 2024-05-01 - DOM XSS and Path Validation
**Vulnerability:** The `viewer.html` script read `window.location.hash` directly without validation, passing it into sensitive contexts like file fetching (`fetch()`) and link targets (`rawLink.href`). This created a DOM XSS vulnerability if an attacker used `#javascript:alert(1)` or similar payloads, as well as an Open Redirect/Arbitrary Fetch via URL protocols like `https://`. Furthermore, markdown contents rendered client-side with `marked.parse` were inserted directly into the DOM using `innerHTML` without sanitization, risking stored or reflected XSS if malicious markdown was supplied.
**Learning:** `window.location.hash` is user-controlled input and must be strictly validated. Checking for dangerous patterns (`..`, `//`, `:`) and ensuring local paths (starting with `/`) prevents traversal and open redirects. Client-side markdown parsers typically return raw HTML by default, requiring a separate sanitization step (e.g., using `DOMPurify`) before DOM insertion.
**Prevention:** Always sanitize input derived from URLs before using it in fetch requests or attribute bindings. Wrap any markdown rendering that is injected into the DOM with a robust HTML sanitizer like DOMPurify.
